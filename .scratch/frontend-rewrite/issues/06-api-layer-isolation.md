# API layer isolation

Type: grilling
Status: resolved

## Question

Design the seam that isolates mock/prod switching out of the "real" API layer. Current state: `lib/platformApi.ts` (1,504 lines) directly imports mock data from `mockPlatformApi.ts`, gated by a runtime `import.meta.env.VITE_USE_INSIGHTS_PLATFORM_API` flag baked into the file itself — mock/prod switching logic lives inside the "real" API layer instead of being isolated behind an interface.

Decide the target interface/adapter shape so mock and prod implementations sit behind a clean boundary — consult `/codebase-design` for the seam/interface vocabulary.

## Answer

**Revised — the initial resolution undersold this.** A 1,504-line file implementing a flat 51-method interface across roughly ten unrelated business domains (search, candidates, job postings, job matching, parsing, insights, admin/tenant provisioning, ops health, platform config) is a God Object, full stop — not a defensible interface with a couple of composition bugs. Single Responsibility is violated (job-matching changes and admin-provisioning changes touch the same file), Interface Segregation is violated (every consumer depends on all 51 methods even when it calls 3), and nothing bounds the file's growth. That reaction — "an engineer would run away from this" — is correct.

**Why React Query makes the whole abstraction mostly unnecessary**: with React Query as the mandated data-fetching layer ([Architecture & state-management conventions](05-architecture-state-conventions.md)), each feature should own small, typed query/mutation hooks calling a thin shared transport directly — not a giant pre-built interface enumerating every possible call. The only genuinely cross-cutting piece is the low-level transport (`invokeFunction`/`invokePlatform`: auth headers, error normalization) — that stays small and shared. Everything above it decomposes by feature.

**The refactor**:

- `lib/api/client.ts` — the *only* shared piece. The low-level transport (`invokeFunction`, `invokePlatform`), auth header attachment, error-shape normalization. Small (~100-150 lines) and stays that way, because nothing domain-specific belongs in it.
- **The 51-method `PlatformApi` interface is deleted**, not kept. Each feature gets its own `features/<domain>/api/` folder (already part of the three-tier structure from ticket 05) holding narrow React Query hooks that call the shared client directly — e.g. `features/jobs/api/{useJobPostings,useJobPosting,useSaveJobPosting,useJobMatchingRuns}.ts`, `features/search/api/{useSearch,useCompare}.ts`, `features/insights/api/useInsightsDashboard.ts`, `features/admin/api/useTenants.ts`. Response-shape mapping moves alongside each feature's own hooks — a feature only knows about its own slice, never the other nine — but see below, "mapping" is replaced wholesale, not just relocated.
- **Response handling — feature-owned wire schemas and compatibility adapters replace all 12 hand-rolled fallback mappers.** The shared `lib/api/client.ts` returns successful JSON as `unknown` and stays domain-blind; it does not expose `invoke<T>()` or recursively camel-case keys. Under each feature's `api/` folder, private Zod wire schemas describe the backend payloads that actually exist today, including verified legacy snake_case/camelCase variants. A named adapter parses that wire shape and transforms it into one canonical camelCase schema whose `z.infer` type is the only type exported to React Query, hooks, forms, and components. Request encoders perform the reverse translation for writes. Validation and transformation happen inside the query function before data enters React Query's cache, never in a component or React Query `select`.

  The compatibility layer is strict about what it tolerates. Before replacing a mapper, inventory every owned alias chain (for example `current_title ?? currentTitle ?? job_title ?? title`), verify each accepted variant against backend source or a captured real payload, and record precedence plus null/absence semantics. If multiple aliases are simultaneously present with conflicting non-null values, fail parsing rather than choose silently. Required and security-sensitive fields never receive defaults or broad `String()`/`Number()` coercion. Each accepted variant gets a fixture-based adapter test, plus rejection tests for missing required fields, invalid values, and alias conflicts. This preserves known current backend variation without preserving speculative hedging.

  This was a gap in the first pass of this ticket, caught on review. The current `platformApi.ts` and feature mappers take `unknown` and use `String(record.x ?? record.y ?? default)` chains via `asRecord()`; malformed fields silently become defaults, including one case where a missing `role` becomes `"owner"`. Zod now validates rather than silently manufacturing data, while the explicit adapter contains the unavoidable legacy naming. When the backend is unified later, only wire schemas, transforms, request encoders, and their fixtures change; the canonical frontend model and every caller remain stable. Form and canonical domain schemas may share field fragments where their semantics match, but a wire response schema is never reused as a form schema merely because some keys overlap.
- **Mocking moves to MSW** (confirmed against current docs — network-level request interception, described by its own docs as the industry-standard approach for exactly this). Handlers are colocated per feature and composed into one set; `setupWorker` activates them in dev behind an explicit flag, `setupServer` activates them in Vitest (ties directly into [Testing, lint, prettier & build tooling](04-testing-lint-prettier-build-tooling.md)). Because interception happens at the network level, **no hook or component is ever mock-aware** — the same code runs against real or mocked calls, which also fully retires the original `createLazyMockApi()` Proxy hack and the "real implementation quietly falls back to mock" problem in one move, rather than needing a separate composition-root fix for it.
- The backend transport-fallback question (Edge Function vs. direct RPC, confirmed still live by checking `supabase/functions/platform/index.ts` and recent migration dates) stays exactly as scoped before: not replicated in the frontend, left as a concern for the backend's own future rewrite (see map's Out of scope).
- The current per-method guard (`if (!supabase || !tenantIds?.length) return mock.xxx(...)`) goes away structurally — "no tenant selected, show empty state" becomes explicit application logic in the relevant feature hook, not a reach into mock data.

**`lib/api/json.ts` in full — none of it survives.** `asRecord`/`asArray`/`toStringArray`/`toNumber`/`nullableString` are superseded by endpoint-specific Zod wire schemas and explicit transforms. The remaining `errorMessage(error: unknown)` is an error-display concern and is superseded by [Error handling & observability strategy](07-error-handling-observability.md), not carried forward as parsing infrastructure.

New dependencies: `msw`, plus `zod` (already added in ticket 05 for forms — reused here, not a second new dependency). No `PlatformApi`-shaped file, and no hand-rolled `map*`/`json.ts`-style runtime coercion, survives into `frontend-v2`.
