# Frontend Architecture Decisions

## Destination

All foundational architecture decisions for the frontend rewrite are locked and recorded as ADRs: rewrite strategy, UI component library & styling foundation, design token system, testing/lint/prettier/build tooling, architecture & state-management conventions, API layer isolation, error-handling/observability strategy, and routing/recruiter-candidate structure. Nothing gets implemented as part of this map — it ends when every ticket is closed and each decision is documented with its reasoning, ready to hand to `/to-spec`.

## Notes

- **Master operating stance, above every other note below**: treat the current implementation as disposable, not the product users recognize. For an existing screen, carry forward its business capability and visible CSS layout — panel and control placement, proportions, spacing rhythm, section composition, colors, and button feel — while rebuilding the implementation on the decided stack (React Query + zod + shadcn + React Hook Form + TanStack Table + Zustand where scoped). Never preserve an old file structure, helper, store, state-management pattern, CSS/class-name implementation, parser, or coercion utility merely because it already exists. This distinction is the default lens for every remaining ticket and every future `/implement` session off this map.
- **Do not wait for backend return types to be unified.** When a response shape is unclear, inspect the producing Edge Function/RPC and capture representative real payloads. Each feature owns private wire schemas and explicit compatibility adapters for verified current variants; the shared transport returns `unknown` and performs no global casing conversion. Adapters transform to one canonical camelCase frontend model before React Query caches data. Every existing alias chain is inventoried with precedence, null, and conflict behavior and covered by fixtures; hypothetical variants are rejected. Later backend cleanup changes only those wire schemas/transforms/request encoders and their tests.
- **The `react-best-practices` skill is mandatory for every `/implement` session.** Invoke it at least once when building any part of `frontend-v2/` and follow it. Don't copy its content into tickets — it's an always-available skill. Two reconciliations (checked, no real conflict): where it says **SWR**, we use **React Query** (same purpose, decided tool — don't switch); and its server-rendering / Next.js rules don't apply (we're a client-side Vite SPA — its `next/dynamic` = our `React.lazy`, skip the SSR-only rules). Everything else agrees with or reinforces the spec. Scaffold step must make the skill available from `frontend-v2/`.
- **`gaps-and-recommendations.md` and `additional-scan-findings.md` are binding addenda, accepted as-is.** They carry file-level detail beyond the tickets and lock the compatibility-adapter policy, file/asset naming, repo-wide inline-style/color bans, dependency cleanup, stable list keys, and accessible destructive confirmation. All of it is folded into `spec.md`; where an addendum is more specific, its specificity governs.
- Consult `/grilling` and `/domain-modeling` for every ticket by default; `/codebase-design` additionally for [Architecture & state-management conventions](issues/05-architecture-state-conventions.md) and [API layer isolation](issues/06-api-layer-isolation.md); `/research` for any ticket needing external docs (e.g. the Tailwind v4 migration guide, shadcn conventions).
- Standing preference: UI library = shadcn/ui on Tailwind's latest major (v4). [UI library & component approach](issues/02-ui-library-component-approach.md) confirms and details this — it does not re-litigate the choice from scratch.
- Standing preference: client state = lean toward Zustand for cross-cutting UI state. [Architecture & state-management conventions](issues/05-architecture-state-conventions.md) confirms and scopes this — it does not re-litigate the choice from scratch.
- Vite upgrade (6.4.3 → latest, corrected mid-effort from v7 to **v8** since Vite released v8 while this map was in progress — includes the Rolldown bundler merge) is folded into [Testing, lint, prettier & build tooling](issues/04-testing-lint-prettier-build-tooling.md), not its own ticket.
- The outer-repo CI pipeline is explicitly out of scope. [Testing, lint, prettier & build tooling](issues/04-testing-lint-prettier-build-tooling.md) is scoped to this project's own tooling only.
- The rewrite happens in a fresh sibling directory in this repo (working name: `frontend-v2/`), not a separate repo or the existing `frontend/` in place. See [Rewrite strategy](issues/01-rewrite-strategy.md).
- Standing constraint: this is a re-implementation, not a redesign — the current app's **screen layouts, core brand colors, and button look/feel carry over**, but they are recreated with the modern component/token system rather than literal CSS, class names, or 219 hardcoded values. Applies directly to [Rewrite strategy](issues/01-rewrite-strategy.md), [UI library & component approach](issues/02-ui-library-component-approach.md), and [Design token system](issues/03-styling-system-token-rebuild.md).
- **Visual parity describes the rendered result, not the implementation.** Match where panels, controls, and actions appear and how the screen is composed. Do not carry over inline styles, magic numbers, ad hoc breakpoints, state management, helpers, or parsing code. Use the old screen as visual evidence and the new stack as the implementation authority. Applies to every remaining ticket.
- **"An interface exists" is not the same as "the structure is fine."** [API layer isolation](issues/06-api-layer-isolation.md)'s first pass called a 1,504-line, 51-method God Object mostly acceptable because it had a well-typed interface — wrong, and corrected after pushback. Every remaining ticket should ask not just "does this have a type/interface" but "is this the right shape and size at all" before accepting existing structure.
- **Strict TypeScript, no exceptions**: no `any`, no `as` casting (current codebase: 169 `as` casts, 7 of them `as unknown as`/`as any` specifically bypassing the type system). When a type isn't obvious, check the actual package's shipped types (its `.d.ts`/exports) rather than guessing or reaching for `any`. Don't write an explicit type annotation where the compiler already infers it correctly — idiomatic modern TypeScript, not maximal annotation. See [Testing, lint, prettier & build tooling](issues/04-testing-lint-prettier-build-tooling.md) for the mechanical enforcement (lint rules + tsconfig flags). Applies to every ticket's resulting code and to all future `/implement` work off this map, not just the tooling ticket.
- **Never hand-parse an external value — validate it.** API responses use private wire schemas plus explicit transforms into canonical frontend schemas; known legacy aliases are allowed only when verified and fixture-tested. No global casing conversion, broad coercion, silent default, or raw wire type may cross into React Query or UI code. The same validation rule applies to `localStorage` and URL params.
- Full responsive support is in scope — mobile through desktop, not desktop-primary. See [Design token system](issues/03-styling-system-token-rebuild.md) for the fluid-typography/breakpoint approach this implies.
- New standing dependencies decided in [Architecture & state-management conventions](issues/05-architecture-state-conventions.md): React Hook Form + Zod (all forms, via shadcn's native `Form` primitives) and TanStack Table (all data tables, via shadcn's `Table` markup). Relevant to [Testing, lint, prettier & build tooling](issues/04-testing-lint-prettier-build-tooling.md) (new deps to account for) and [API layer isolation](issues/06-api-layer-isolation.md) (zod schemas as a possible shared contract at the API boundary).
- Already-obvious fixes that matter to v2 (raw error text, missing `alt`, native destructive confirmation, unstable list keys, duplicate lockfiles, and missing README) are implementation requirements in the build tickets and standing rules; they need no further architecture decision.

## Decisions so far

- [Rewrite strategy: strangler vs. parallel](issues/01-rewrite-strategy.md) — full parallel rewrite from scratch, in a fresh `frontend-v2/` directory in this repo (not strangler, not a separate repo); core brand colors carry over via a proper token scale (not the 219 raw hex values as-is), this is a re-implementation not a redesign.
- [UI library & component approach](issues/02-ui-library-component-approach.md) — shadcn/ui on Tailwind v4 (via `@tailwindcss/vite`), confirmed; component mapping settled (Button, Dialog, AlertDialog); the 6+ dropdown/picker implementations consolidate into one shared combobox component instead of staying separate. Generated shadcn primitives stay aligned with upstream; product identity comes from semantic tokens and shared composites, and recurring semantic differences use variants rather than one-off feature classes.
- [Design token system](issues/03-styling-system-token-rebuild.md) — colors extracted from actual usage (background/primary/success/warning carried over, new `--destructive: #f87171`), custom neutral scale anchored on `#39393a` (not Tailwind's stock gray), chart colors kept separate (`--chart-1`…`--chart-8`), Inter self-hosted with a restrained fluid type scale, and explicit 6/8/12/18/22px radius roles plus a 16px button radius. The neutral elevated shadow is reserved for overlays. A standard 5-step breakpoint scale replaces the current ad hoc breakpoints. The z-index scale is eliminated outright, not migrated — Radix's portal-based stacking (via shadcn) makes it unnecessary.
- [Architecture & state-management conventions](issues/05-architecture-state-conventions.md) — `screens/` dropped, three-tier component structure (`components/ui/`, `components/`, `features/<domain>/{pages,components,hooks,api,types}`); container/presentational split formalized; React Query mandatory for all server state; React Hook Form + Zod mandatory for all forms; TanStack Table mandatory for all data tables; `@/` alias mandatory everywhere. **Amended (gaps review): state-ownership reversed** — filters/sort/pagination/tab → URL `searchParams` (not Zustand), shortlist → React Query (server data, not Zustand); Zustand shrinks to just sidebar-collapse + transient Compare selection. Types derived from zod (`z.infer`); small rebuilt auth/tenant-scope context with `scopeKey`-prefixed query keys; three guarded route groups (public/auth/admin); all data through Edge Functions (no direct table reads); Recharts via shadcn chart; per-route lazy loading.
- [Testing, lint, prettier & build tooling](issues/04-testing-lint-prettier-build-tooling.md) — Vitest + React Testing Library, coverage by convention not percentage gate; ESLint flat config with type-aware linting plus `react-hooks`/`react-refresh`/`jsx-a11y` plugins (the last directly targets the original audit's accessibility gaps); Prettier reconciled via `eslint-config-prettier`; npm confirmed as the standard package manager (fact, not decision — `pnpm-lock.yaml` deleted); Vite upgraded to **v8** (corrected from v7, see Notes). Amended: `no-explicit-any`/`consistent-type-assertions`/`no-unnecessary-type-assertion`/`no-inferrable-types` all set to `error`, plus `noUncheckedIndexedAccess` in `tsconfig.app.json` — no `any`, no casting, current codebase has 169 `as` casts to eliminate.
- [API layer isolation](issues/06-api-layer-isolation.md) — **revised to isolate current backend inconsistency without waiting for backend cleanup**: the 51-method `PlatformApi` God Object is deleted and replaced with a domain-blind transport returning `unknown`, plus feature-owned wire schemas, compatibility adapters, request encoders, and React Query modules. Only canonical camelCase schema-derived models enter React Query or UI code. Verified current aliases are explicitly inventoried and fixture-tested; conflicting aliases and malformed/security-sensitive data fail instead of defaulting (especially no missing `role` → `"owner"`). Mocking moves to MSW, and backend Edge-Function/RPC fallback logic remains un-replicated.
- [Routing structure: clean URLs & recruiter/candidate naming separation](issues/08-routing-and-recruiter-candidate-separation.md) — new ticket, not in the original seven. Confirmed **not** two apps (only 2 of 20+ routes are public/candidate-facing; the router already separates them structurally) — the confusion is a naming collision, "Candidate" overloaded to mean both the recruiter's data record and (implicitly) the job seeker. Fix: reserve "Candidate" for the recruiter-side entity, give the public surface its own term and its own `features/careers/` folder, no shared naming. Separately: switch from hash routing (`/#/candidates`) to clean URLs (`createBrowserRouter`) — requires an `.htaccess` SPA-fallback rewrite rule on cPanel at actual deploy time (hash routing was deliberately chosen to avoid exactly this), flagged as a required follow-up for whoever executes the cutover.
- [Error handling & observability strategy](issues/07-error-handling-observability.md) — scope was broader than cited (7 files leak raw `error.message`, not 3). Two-tier `react-error-boundary` boundaries (router-level + per-route); shadcn `Alert` for blocking errors, Sonner for transient ones (shadcn's own `Toast` is deprecated); a message-mapping layer so raw exception/zod-parse text never reaches the UI. Error-reporting tool (Sentry etc.) explicitly deferred, not added now — pre-launch, no error volume yet.

## Out of scope

- **Backend rewrite.** The backend (`supabase/functions/`, RPCs) is acknowledged as its own spaghetti needing a full rewrite, but that's a separate future effort. Surfaced while resolving [API layer isolation](issues/06-api-layer-isolation.md); this map's frontend API layer is deliberately designed to stay pluggable so it can be repointed at whatever the backend becomes later, rather than the frontend rewrite trying to fix backend architecture now.
