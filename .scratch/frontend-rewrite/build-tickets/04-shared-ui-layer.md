# 04 — Shared UI layer (shadcn primitives + domain composites)

**What to build:** the reusable building blocks every screen uses, built once and styled from the tokens. This kills the 144 copy-pasted buttons, the 9 copy-pasted modals, and the 6+ copy-pasted dropdowns. After this, a screen composes from these pieces and never hand-writes styling.

**Blocked by:** 03 (the pieces need the tokens).

**Status:** ready-for-agent

**References:** `../issues/02-ui-library-component-approach.md` (the decision), `../spec.md`, `../gaps-and-recommendations.md` Part 3 §B, `../additional-scan-findings.md` Parts 2–5.
**Standing rules:** see ticket 01. Extra rule for this ticket: keep generated shadcn primitives aligned with upstream. Product identity belongs in semantic tokens and domain styling belongs in shared composites; recurring semantic differences use `variant`/`size` props. No one-off feature-specific CSS names, hardcoded colors, or inline `style` except genuinely dynamic values passed through a CSS custom property where practical.

- [x] shadcn primitives installed and owned: Button, Dialog, AlertDialog, the one shared searchable-select (Popover + Command combobox), Form primitives, Table markup, Alert, Sonner Toaster, Card, Badge, Progress, Avatar, Skeleton, and the chart (Recharts) wrapper
- [x] Domain composite components built on top of those primitives, variant-driven: StatCard, ScorePill, TenantBadge, EmptyState, PageHeader (map the old `ui.tsx` grab-bag: Panel→Card, Tag→Badge, ProgressBar→Progress, Avatar→Avatar, EmptyState→compose on Card)
- [x] The single shared combobox replaces all per-domain dropdown/picker logic (skills, location, company, seniority, etc.) — parameterized by data source, not reimplemented per domain
- [x] Familiar interface icons come from `lucide-react`; custom image/SVG assets are limited to verified brand or domain artwork with no Lucide equivalent
- [x] Every primitive is keyboard-accessible and screen-reader-correct out of the box (via Radix) — no `onClick` on non-interactive elements
- [x] Each shared component has at least a basic render test where it carries real conditional logic
- [x] No hardcoded colors, spacing, or z-index values — everything reads from tokens

## Comments

Implemented on branch `frontend-refactor`.

**Primitives** — installed via `npx shadcn add` (this shadcn version, 4.13.1/"radix-nova"
style, differs from the classic community CLI in a few ways worth noting for later tickets):
Button (already existed from ticket 01), Dialog, AlertDialog, Popover, Command, Table, Alert,
Sonner, Card, Badge, Progress, Avatar, Skeleton, Chart. "Form primitives" resolved to this
shadcn version's actual current offering — the classic RHF-bound `Form` component doesn't
exist in this registry (`shadcn add form` silently produces no files); the modern replacement
is the `Field`/`FieldLabel`/`FieldError`/etc. family (RHF-agnostic, `errors` passed as a
plain prop), which pulled in `Label`, `Separator`, `Input`, `Textarea`, `InputGroup` as
registry dependencies. No `react-hook-form`/`zod` dependency was added — no real form exists
yet to wire one up against; that's for whichever ticket builds the first actual form.

**Vendored files stay vendored** — the CLI-generated primitives violated this repo's strict
lint/format rules in ~30 places (mostly `chart.tsx`, wrapping Recharts' inherently
loosely-typed payload/tooltip API). First pass hand-fixed each one; on review, that was the
wrong call — `src/components/ui/**` gets regenerated by `npx shadcn add`, so hand-fixes there
are lost work the next time a component is re-added or upgraded, for no real benefit (this is
library code we don't hand-maintain, not house code). Reverted every file in that directory
back to exact stock CLI output (including `next-themes` in `sonner.tsx`, `key={index}` in
`chart.tsx`/`field.tsx`, etc.) and instead extended the existing `src/components/ui/**`
override in `eslint.config.js` (ticket 01 already exempted that directory from the
filename-naming/`react-refresh` rules for the same "vendored, not house style" reason) to
also turn off the strict-TypeScript/no-array-index-key/no-inline-style/jsx-a11y rules that
don't make sense for wrapped third-party library internals. Added the matching exemption to
`.prettierignore` too, mirroring the existing `.claude`/`.agents` vendored-content entry.
Going forward: only touch a file under `src/components/ui/` for an actual behavioral reason,
never just to satisfy lint/format.

**Test environment gaps** — jsdom implements neither `ResizeObserver` nor
`Element.scrollIntoView`, both of which `cmdk` (Command's underlying library) calls
unconditionally. Added minimal no-op polyfills to the shared `src/test/setup.ts` (not a
per-test hack) since this affects every future test touching `Command`/`Combobox`. Radix's
`Avatar` `Image` has a similar issue (probes load status via a real `new window.Image()`,
which jsdom never resolves) — stubbed locally in `TenantBadge.test.tsx` only, since that one
is narrower and less obviously reusable elsewhere yet.

**StatCard's dead `tone` prop** — the old `StatCard`'s `tone` prop (`primary`/`secondary`/
`tertiary`) was already called with different values at real call sites (e.g.
`ManatalSyncStatusPage.tsx`) but had **no corresponding CSS** — `grep` of `primitives.css`
confirms only `.stat-card`, never `.stat-card--primary/secondary/tertiary`. Kept the prop
(callers already depend on its existence) but gave it real meaning via the icon accent chip's
color, using tokens from ticket 03 (`primary`/neutral/`success`) — a bug fix using existing
tokens, not new invented styling.

**StatCard's `loading` prop** — new API surface, not on the old component itself, but a
direct consolidation of a real hand-copied pattern: the old app built the same skeleton shape
(`.stat-card--loading` + 4 `.stat-card__skeleton*` bars) independently at 5+ call sites
(`ParsingOverviewPage.tsx`, `PlatformAdminDashboardPage.tsx`,
`ParsingDetailSkeleton.tsx`) by reaching past `StatCard` entirely and hand-building a raw
`Panel` with that class. Giving `StatCard` a `loading` prop that renders shadcn `Skeleton` in
each slot is exactly the "kills copy-pasted X" this ticket's opening line describes, using a
primitive (`Skeleton`) this ticket already requires installing.

**Combobox** — one component

**`MetricBars` not built** — `gaps-and-recommendations.md` §B lists it alongside `StatCard`/

**Combobox** — one component (`ComboboxSingleProps | ComboboxMultipleProps` discriminated on
`multiple`) replacing both `PickerDropdown` (single-select) and `FilterMultiSelect`
(multi-select with chips + freeform creation), built on Popover+Command per the decision doc.

**Follow-up: the visual pass above found real bugs.** Playwright (headless Chromium, cached
locally) was used to actually render the app and screenshot it, since the Chrome extension
still wasn't available. Two things were wrong with `Button`, both now fixed in `button.tsx`
and `index.css`:
- No `cursor-pointer` on the base `cva` class — this shadcn style genuinely omits it by
  default. Added unconditionally (a `<button>`'s native default is `cursor: default`, not
  `pointer`, in most browsers).
- The default/lg sizes rendered as a full pill, not the old app's moderate rounded
  rectangle. Root cause was two-fold: (1) ticket 03 only carried over the old app's *panel*
  radius (22px, `--radius`) into `--radius-lg`, missing that the old app used a **separate,
  smaller 16px radius for buttons specifically** (`shell.css:132`) — added a dedicated
  `--radius-button: 16px` token and pointed `button.tsx` at it instead of `rounded-lg`; (2)
  even at 16px, shadcn's compact default height (`h-8`, 32px) is less than 2×16, so it was
  still a pill — the old app's buttons were actually ~48-50px tall (`padding: 0.85rem 1rem`
  around an inherited 16px font, no explicit height). Bumped `default` to `h-11`/`text-base`
  and `lg` to `h-12`, matching that proportion; left `xs`/`sm`/`icon-xs`/`icon-sm` as
  compact options the old app never had (no old equivalent to preserve for those).

This was caught by actually rendering the app, not by reading CSS — the case study for why
ticket 01's new "Layout parity" standing rule requires comparing a real render against the
old screen before marking a ticket done, not just checking the prose checklist.

**Second follow-up: primary button hover state, button font-weight, and a non-fluid font size,
all from direct user feedback.** Three real problems in `button.tsx`, none caught by the first
visual pass:

- **`default` variant's hover made it look disabled, not interactive.** It was
  `bg-primary hover:bg-primary/80` — dropping opacity over the dark `--background` reads as
  *desaturating toward gray*, not brightening. Every other variant (`secondary`, `outline`,
  `ghost`, `destructive`) moves *toward* more saturation/prominence on hover; `default` was the
  one exception moving the opposite direction, which breaks the basic hover-affordance
  expectation (hover should invite the click, not look like the control just got disabled).
  First attempt restored the old app's literal primary-button identity — a diagonal
  `--primary`→`--primary-strong` gradient plus a primary-tinted shadow that grows on hover
  (`shell.css:148`'s treatment, rebuilt from tokens already in the system, no new color). **User
  feedback rejected this direction outright**: "the glow is not required at all... I want
  something elegant basic like material design or the default shadcn" — explicitly not a
  request to reproduce the old app's specific visual treatment, just its base color. Landed
  instead on stock shadcn's own `default`-variant shape applied to our tokens: flat `bg-primary`,
  a plain (non-colored) `shadow-xs`, and a shallower hover drop (`hover:bg-primary/90` instead of
  `/80`) — enough to read as a state change without the muddying `/80` caused. No gradient, no
  colored shadow, no lift transform. **Note for future agents: don't reintroduce the
  gradient/colored-shadow treatment here** — it was tried and explicitly turned down for this
  component, not merely untried.
- **Base button font-weight read as too heavy.** The shared `cva` base applied `font-medium` to
  every variant/size uniformly; direct feedback ("the font is too much bold") brought it down to
  `font-normal` across the board — this is a base-class change, not a variant override, since the
  complaint wasn't scoped to one variant.
- **`sm` size used `text-[0.8rem]`, a hardcoded value outside the fluid type scale** — every
  other size reads a real `text-*` token (`text-xs`, `text-base`), which resolves to ticket 03's
  `clamp()`-based fluid scale and grows with viewport; `sm` alone was frozen at a fixed 12.8px,
  silently opting out. This is very likely what prompted the original "buttons don't look like
  they use the fluid font scale" report — the default/lg sizes were already fluid (verified via
  Playwright: computed `font-size` moves 16.12px → 17.69px across viewport widths), but `sm` was
  not, so it read as inconsistent. Switched to the `text-sm` utility, which resolves to the
  existing `--text-sm` token instead of a new hardcoded number.
