# 05 — API transport + auth/tenant-scope + React Query client

**What to build:** the one door to the backend and the login/company logic behind it. All data goes through one small transport. The app knows who the user is, whether they're an admin, and which company they're viewing — and switching company automatically reloads the right data. No fake data path anywhere.

**Blocked by:** 01.

**Status:** ready-for-agent

**References:** `../issues/06-api-layer-isolation.md` and `../issues/05-architecture-state-conventions.md` (decisions), `../spec.md`, `../gaps-and-recommendations.md` Part 2 §1–3 and §13, `../additional-scan-findings.md` API compatibility decision.
**Standing rules:** see ticket 01. Note especially: no mock/real switch, no `if (!supabase)` branch, no silent fallback, no global casing converter — one real transport path and feature-owned compatibility.

- [x] One small transport wraps Edge Function invocation and normalizes the error shape; successful JSON is `unknown`, with no generic `<T>` cast, domain schema, recursive casing conversion, broad coercion, or fallback value
- [x] Establish the per-feature adapter template: private wire schemas describe verified current backend payloads; named parse/transform functions produce canonical camelCase schemas/types; request encoders translate canonical command/form input to current backend request keys
- [x] Parsing and transformation occur inside each React Query query/mutation function before data enters the cache. React Query `select` may derive views from canonical data but never validates or normalizes wire payloads; raw wire keys/types are not exported from `api/`
- [x] Create a durable compatibility inventory, grouped by endpoint/operation, seeded from every current response alias/fallback chain. For each canonical field it records accepted wire names, evidence (backend source or captured real response), precedence, required/nullable/optional semantics, and retirement notes; speculative aliases are not accepted
- [x] Compatibility-inventory completeness is checked before deleting an old mapper: inspect every current `map*`/`apiMappers` response path and statically search its `??`, `||`, ternary, casing-alias, coercion, and default expressions; classify each as a verified wire alias, request encoding/default, product-level presentation fallback, or dead behavior. No owned mapper is retired until every retained response case has a fixture
- [x] Alias resolution is fail-loud: if two aliases for one canonical field are present with conflicting non-null values, parsing fails; missing required/security-sensitive values and invalid enums/ranges fail; no `String()`/`Number()` blanket coercion or privilege-bearing defaults
- [x] Adapter contract tests use raw fixtures for every accepted wire variant and assert exact canonical output, plus missing-required, invalid-value, null-semantics, and conflicting-alias rejection cases; one representative MSW + React Query test proves only canonical data reaches the cache/hook
- [x] All data goes through Edge Functions; no direct database-table reads from the frontend. Where a needed Edge Function is missing, flag it as a backend task — never re-add a direct table read as a workaround
- [x] A small auth context: session, user, memberships, isPlatformAdmin, currentTenant, selected tenant, and scope mode — rebuilt lean, no redundant state-sync or manual token juggling
- [x] Scope mode (single company vs. admin "all companies") + selected tenant are one concrete auth-preferences record in the auth context, persisted through an exact versioned Zod schema; one `useTenantScope()` hook returns the resolved tenant ids and a stable `scopeKey`
- [x] React Query client set up; the key convention is `[domain, scopeKey, ...params]` so switching company/scope refetches automatically
- [x] The blunt "clear the whole cache on any auth change" is gone; full cache clear happens on sign-out only (a security requirement)
- [x] If required config is missing in dev, the groundwork exists for a clear "app is not configured" state (wired into the shell in ticket 06); never a silent fallback

## Comments

Implemented on branch `frontend-refactor`. Notable deviations/decisions made during implementation:
- The compatibility inventory (checklist items 4–5) was built as three parallel domain audits — `frontend-v2/docs/api-compatibility-inventory/{candidates-search,jobs-insights,admin-ops-parsing-misc}.md`, plus a `README.md` index — statically cross-checking every `??`/`||`/casing-alias/coercion/default expression in the current `frontend/` mappers against actual backend source (Edge Functions + SQL migrations). Headline finding: nearly every alias in the current frontend is speculative/dead — for a given field, the backend has only ever emitted one casing/name, not the multiple the old code defensively hedges against. A few real findings surfaced along the way (a `candidate_detail` select-list/mapper field mismatch that needs a live-payload check before that feature's adapter ships; the known `role` → `"owner"` silent-default bug; several other silent-default/blind-cast patterns) — flagged in the docs for the tickets that build those features, not fixed here (out of scope for ticket 05, which only builds the transport + auth/tenant-scope + the adapter template).
- The adapter template (item 2) is documented at `frontend-v2/docs/api-adapters.md`, with `@/lib/auth/api/authContext.ts` as the concrete reference implementation (wire schema → parse/transform function → canonical schema). That endpoint happens to have zero real alias/fallback chains once cross-checked against the backend, so it doesn't exercise the union/conflict-resolution machinery described in the doc — future feature tickets (candidates, jobs, search) are where that part of the template actually gets exercised, guided by the compatibility inventory.
- `AuthContext`/`useAuth` were split out of `AuthProvider.tsx` into their own `authContextStore.ts` (no JSX) purely to satisfy `react-refresh/only-export-components` — a provider component and its consumer hook living in the same file is otherwise the natural shape.
- Added a `src/vite-env.d.ts` opting into Vite 8's `ViteTypeOptions.strictImportMetaEnv`, since the default `ImportMetaEnv` has a `[key: string]: any` fallback that made every `import.meta.env.VITE_*` read `any` and tripped `no-unsafe-*` lint rules.
- `VITE_API_BASE_URL` (present in `frontend-v2/.env.example`/`README.md` since ticket 01, and still real for the old `frontend/`'s hand-rolled fetch transport) is dropped from `frontend-v2`'s copies: the new transport uses `supabase.functions.invoke()`, whose target URL is derived solely from `VITE_SUPABASE_URL` with no override hook, so documenting the var here would be a dead/silently-ignored setting. The shared repo-root `.env.example` is untouched — the old app still reads it.
- A minimal test kit was laid down here rather than in a dedicated tooling ticket (none exists in `build-tickets/`): `src/test/msw/{handlers,server}.ts` (the one network seam), `QueryClientTestProvider` for hook tests, and `TestAuthProvider` plus `createTestAuthContextValue` — the "one honest exception" from the spec's testing section: auth is injected directly into `AuthContext` rather than intercepted via MSW, since it's a stateful client MSW can't cleanly intercept. A router-inclusive `renderWithProviders` is left for ticket 06, once there's an actual router to wrap.
- `lint`, `test` (48 tests), and `build` all pass.
