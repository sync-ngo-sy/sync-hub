# 01 — Project scaffold + tooling

**What to build:** a runnable, empty `frontend-v2/` project with all quality tooling in place, so every later ticket has a solid base. A developer can clone, install, run the dev server, and run lint/test/build — all green — against a project that has no features yet but a fully-configured foundation.

**Blocked by:** None — can start immediately.

**Status:** ready-for-agent

**References:** `../spec.md`, `../gaps-and-recommendations.md`, and `../additional-scan-findings.md` (all binding) in the feature folder.
**Standing rules (apply to every ticket):** invoke the `react-best-practices` skill at least once and follow it (use React Query, not SWR; skip the server-rendering rules — this is a client-side Vite SPA); strict TypeScript (no `any`, no `as` casting); no mocks or fake data in app code; never show raw error text; use stable domain list keys; give meaningful images descriptive `alt` text and decorative images empty `alt`; no hardcoded TS/TSX colors or non-dynamic inline styles. Every backend response follows ticket 05's `unknown` transport → private verified wire schema → explicit compatibility adapter → canonical camelCase schema flow before entering React Query; raw wire keys/types never reach hooks, forms, pages, or components. The same owner-specific Zod rule applies to local storage, URL state, and MSW request bodies: never add generic record/coercion helpers or manually probe unknown properties. Before extracting a shared helper, identify two current v2 consumers and confirm that the selected platform or library does not already own the behavior; do not create generic `utils` dumping grounds. Reusable Tailwind utility styles live in `src/index.css`, not exported TypeScript class strings. Keep shadcn source unchanged unless a ticket requires a real behavior or accessibility fix. Use semantic HTML for structural regions (`header`/`nav`/`main`/`footer`/`section`/`article`/`aside`, not `div` soup) — this applies to every page and to the app shell itself, not just forms/landmarks called out elsewhere. **Layout parity:** when a ticket ports a screen with an old-app equivalent, look up its real file in `screen-inventory.md`'s `Screen (file)` column, read it, and reproduce its actual visible layout (panel/control placement, positioning, spacing rhythm, section composition — not literal old CSS/hardcoded values) with the new component/token stack; before marking the ticket done, render the new screen (dev server or a screenshot) and compare it side-by-side against the old one, not just against the ticket's prose checklist — see `issues/01-rewrite-strategy.md`'s "Layout, specifically" note for the full reasoning.

## Comments

Implemented on branch `frontend-refactor`. Notable deviations/decisions made during implementation:
- create-vite's newest template ships `oxlint` by default instead of ESLint — removed it and installed the decided ESLint stack instead.
- The default scaffold's `tsconfig.app.json`/`tsconfig.node.json` were missing `strict: true` entirely; added explicitly (plus `noUncheckedIndexedAccess`).
- `shadcn init` misresolved the `@/` alias (tsconfig project-references setup) and wrote files to a literal `./@/` directory; moved them into `src/` and added `paths` to the root `tsconfig.json` so future `npx shadcn add` calls resolve correctly.
- `react-best-practices` was already present under `frontend/.agents/skills/` but had never been mirrored into `frontend/.claude/skills/`, so it wasn't actually invocable — fixed that gap too.
- `check-file` naming rules exempt `src/main.tsx` (Vite's own entry-point convention) and `src/components/ui/**` (shadcn-generated, follows shadcn's own lowercase-filename / co-exported-`variants` convention, regenerated by the CLI).

Ran `/code-review` (Standards + Spec axes) against `main` before committing. Two real findings, both fixed:
- **Standards** — the shadcn-generated `index.css` shipped light-mode as the active default, contradicting the locked "dark-only for now" decision (gaps-and-recommendations.md Part 2 §9). Fixed: `<html class="dark">` + `color-scheme: dark` on `:root`. The token *values* stay the shadcn defaults until the design-token ticket replaces them with real brand colors — only the active/default mode was in scope here.
- **Spec** — (a) `frontend-v2/AGENTS.md`, `docs/agents/`, and the `react-best-practices` skill were originally *symlinked* into `frontend/`'s copies, but those targets are gitignored/untracked there — after a real clone they'd be dangling, so the "reachable from frontend-v2/" criterion wasn't actually durable. Fixed: replaced with real, independent copies inside `frontend-v2/` (`.claude/skills/react-best-practices` is the tracked canonical copy; `.agents/skills/react-best-practices` is a same-directory symlink to it, kept for AGENTS.md-reading tools — note the *repo-root* `.gitignore` ignores `.agents/` everywhere, so that mirror is local-only by long-standing repo convention, same as in `frontend/`). (b) `@fontsource-variable/geist` was shadcn's default font choice, not something ticket 01 or its referenced docs asked for — typography is explicitly the design-token ticket's territory. Removed the dependency and its CSS wiring; `font-sans` now falls back to Tailwind's built-in default until that ticket makes the real call.
