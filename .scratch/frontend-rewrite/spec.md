# Frontend Rewrite: `frontend-v2` Target Architecture

## Problem Statement

The current frontend (`frontend/`, ~37,570 lines across 184 files) has accumulated enough structural debt that it actively slows down and endangers every change made to it:

- The same UI logic is copy-pasted across many files instead of shared: 9 modal implementations share an identical backdrop/Escape-handler string, 6+ dropdown/picker implementations reimplement the same list/search/select logic, and 144 raw `<button>` elements each hand-write their own styling.
- There is no design-token system worth the name: an 18-line token file defines a handful of colors and nothing else, while 219 hardcoded hex/arbitrary-value colors and a 16-value z-index "scale" of magic numbers are scattered through 28 separate hand-written CSS files (5,270 lines).
- There is no test framework, no ESLint configuration, and `npm run lint` is a no-op wrapper around `tsc --noEmit`. Nothing catches unused variables, hook-rule violations, unsafe `any` usage, or dead code.
- The API layer (`lib/platformApi.ts`, 1,504 lines) is a single God Object spanning ten unrelated business domains behind one 51-method interface, with mock-vs-real switching woven into its internals rather than chosen once at composition time, and 12 hand-rolled response parsers that silently default malformed or missing fields — including defaulting a missing `role` field to `"owner"`, the most privileged role in a multi-tenant app.
- There is no error-handling strategy: zero error boundaries exist anywhere, and raw backend/Postgres/Supabase exception text reaches end users directly in at least 7 places.
- Recruiter-facing and candidate/public-facing code and naming are indistinguishable at a glance because the word "Candidate" is used for both the recruiter's data record and, implicitly, the public job-seeker — despite the actual public surface being only 2 of 20+ routes.
- The app is hash-routed (`/#/candidates`), a deliberate historical workaround for cPanel hosting that now just produces worse URLs than necessary.

Together this means: every new feature risks copying an existing anti-pattern forward, nothing mechanically catches regressions, backend errors leak to end users, and the codebase is hard for a new engineer (or an AI agent) to navigate confidently.

## Solution

Rewrite the frontend from scratch in a new sibling directory in this repo (working name: `frontend-v2/`) — a full parallel rewrite, not an in-place strangler migration, given the app is pre-launch with no live users and the owner (solo, no hard deadline) explicitly chose to prioritize doing it right over incremental migration. This is a re-implementation, not a redesign: the current app's core visual identity (brand colors, button look/feel, and each screen's CSS layout) carries forward, distilled into a proper token system rather than copied as literal hardcoded values. Every other technical pattern is in scope to be rebuilt on a modern, coherent stack:

- **UI**: shadcn/ui on Tailwind v4, maximized in coverage (not just the components explicitly named). Keep generated shadcn primitives aligned with their upstream source; product styling comes from semantic tokens and shared composites, with variants used for recurring semantic differences rather than one-off feature classes.
- **Design tokens**: a real token system (legacy-anchored color grades, restrained Inter typography, explicit radius roles, spacing, and breakpoints) derived from the current app's actual usage, not invented fresh and not copied as 219 individual literals. No z-index token scale — Radix's portal-based stacking (via shadcn) removes the need for one.
- **Tooling**: Vitest + React Testing Library, a strict type-aware ESLint flat config, Prettier, npm (not pnpm), Vite v8.
- **Architecture**: a three-tier component structure, formalized container/presentational separation, React Query for all server state, React Hook Form + Zod for all forms, TanStack Table for all data tables, Zustand scoped narrowly to genuine cross-route UI state.
- **API layer**: the 51-method God Object is deleted outright and replaced by a thin shared transport plus small per-feature adapters and React Query hooks. The transport returns `unknown`; feature-owned Zod wire schemas validate the backend shapes that exist today, explicit transforms produce one canonical camelCase frontend model, and only that model enters React Query or UI code. MSW provides network-level mocking.
- **Error handling**: two-tier error boundaries, a consistent user-facing error UI (shadcn `Alert` for blocking errors, Sonner for transient ones), and a message-mapping layer so raw exception text never reaches a user.
- **Routing & naming**: clean URLs (`createBrowserRouter`) instead of hash routing, and an explicit naming/folder separation between the recruiter-facing app (the vast majority of the surface) and the small public candidate-facing surface.

Strict TypeScript discipline (no `any`, no `as` casting, types checked against actual backend/package sources) and a "treat existing code as disposable, understand its purpose, rebuild it properly" stance apply throughout.

## Implementation Decisions

**Rewrite strategy**: Full parallel rewrite, built in a fresh sibling directory in this repo (`frontend-v2/`), not an in-place strangler migration and not a separate repository. Cutover happens later by switching which directory is deployed; scaffolding the new directory is implementation work for the tickets that come out of this spec, not part of this decision. The current app's brand colors, button look/feel, and each screen's CSS layout carry forward — this is a re-implementation, not a redesign — but that constraint is narrow: it does not extend to preserving any existing technical pattern, file structure, helper function, or parsing logic, and it does not mean copying old CSS/class names literally (the 219 hardcoded colors and inline styles are exactly what's being fixed). It means: when a screen has an old-app equivalent, its real layout (positioning, spacing rhythm, section composition) is the reference the new implementation is built to match, using the new component/token stack — see `screen-inventory.md` for the exact old file per screen, and ticket 01's standing rules for how this is verified. Default posture for all other existing code: identify its purpose, discard its implementation, rebuild it on the stack below.

**UI component library**: shadcn/ui on Tailwind v4 (installed via the `@tailwindcss/vite` plugin, not the PostCSS path). shadcn's CLI copies component source directly into the repo (`components/ui/`) and is built on Radix UI primitives, giving keyboard navigation, focus trapping, and ARIA semantics. Generated primitives remain aligned with the upstream shadcn implementation unless a real behavior or accessibility requirement demands a change; ordinary product styling does not justify editing them. Component mapping: raw buttons → shadcn `Button`; the duplicated modals → shadcn `Dialog`; `window.confirm()` → shadcn `AlertDialog`; the 6+ separate dropdown/picker implementations → one shared searchable-select component (shadcn's Popover+Command "combobox" pattern), parameterized by data source, not kept as separate per-domain implementations. Domain-specific composites (e.g. a stat card, an empty state) are built on top of shadcn primitives. Three conventions apply everywhere: (1) use a shadcn primitive when one exists; (2) express global product identity through semantic tokens in `index.css` and recurring domain patterns through shared composites; (3) use `variant`/`size` props for recurring semantic differences, while ordinary page layout remains at the owning presentational component. Do not add feature-specific CSS names or pass arbitrary styling APIs through shared components.

**Design tokens**: a token system built from the current app's actual color usage (extracted via direct inspection, not invented), expressed as CSS custom properties in OKLCH via shadcn's `@theme`/`:root` convention. Core semantic colors (background, foreground, primary, primary-strong, success, warning) carry over from current usage; a `destructive` token is added (previously absent — standardized on the one color already used for a genuine error case in the app, not a newly invented value). A generated neutral scale replaces the ~9 near-duplicate hardcoded dark-gray literals currently in use, anchored on the current dominant background color rather than switched to a generic stock gray scale. Chart/data-visualization colors are kept as a separate token set from UI colors. Inter is self-hosted as the single UI family with a restrained regular/medium hierarchy. Typography uses a compact fluid `clamp()` scale (15→16px body, 30→36px page title) rather than fixed sizes or oversized dashboard display text. Explicit radius roles preserve the legacy soft geometry without multiplying a 22px base: 6/8/12px controls, 18px panels, 22px large containers, and a separate 16px button radius. The existing neutral elevated shadow is reserved for overlays; ordinary hierarchy uses tonal surfaces and borders, never glow or colored shadows. Breakpoints use a standard five-step scale, replacing the current seven ad hoc one-off breakpoint values. No z-index token scale is carried into the rewrite — Radix's portal-based stacking (used throughout via shadcn) removes the class of problem that made a z-index scale necessary in the current app. Full responsive support is required, mobile through desktop, not desktop-primary.

**Testing & tooling**: Vitest + React Testing Library as the test framework, with tests required for business logic (validation schemas, API-layer functions, custom hooks, state stores) as it's written, and not required for purely presentational components without real conditional logic — a convention, not an enforced coverage percentage. ESLint uses a flat config via `typescript-eslint`'s type-aware rule sets, plus hook-rules, Vite-fast-refresh-compliance, and accessibility linting plugins. Prettier is added as the formatter, reconciled with ESLint by disabling ESLint's conflicting stylistic rules rather than running Prettier as an ESLint rule. Strict TypeScript is enforced mechanically: no `any` (as an ESLint rule, not just a style preference), no unnecessary or unsafe type assertions, no redundant explicit type annotations where inference already applies, and an additional strict compiler flag (`noUncheckedIndexedAccess`) beyond the existing base `strict` mode. npm is the standard package manager (an existing fact confirmed from the current repo's actively-updated lockfile and documented workflow, not a new decision); the unused second lockfile is removed. The build tool is Vite, upgraded to its current major version.

**Architecture & state management**: a three-tier component structure — shadcn primitives only in one tier, shared domain-agnostic composite components in a second tier, and per-feature code (`features/<domain>/{pages, components, hooks, api, types}`) in a third — replacing the current split between two parallel, partially-migrated organizational schemes. Every feature's `pages` are thin containers that wire up data (React Query, form state, URL state) and delegate rendering to presentational components. Formatting and transforms stay local until they have independently testable behavior or two real consumers; only then are they extracted into a cohesive, specifically named co-located module. React Query is the only mechanism used for server-state fetching, via hooks in each feature's own `api` folder. React Hook Form plus Zod-schema validation (via shadcn's native form-primitive integration) is the only mechanism used for form state and validation, replacing hand-written `useState`-per-field patterns and hand-written validator functions. TanStack Table (a headless table engine) paired with shadcn's table markup is the only mechanism used for data-table rendering, replacing hand-rolled table logic. An import-path alias is used consistently everywhere; no relative parent-traversal import chains.

**State ownership is fixed** (this supersedes the earlier, narrower Zustand scoping — filters and the shortlist were reassigned during the gaps-and-recommendations review, because the URL satisfies both "filters survive navigation" and "shareable/refreshable URLs" with one mechanism, and the shortlist is genuinely server data): server data → **React Query only** (the shortlist included — it is persisted in the backend, so it is a React Query resource, not client state); filters, sort, pagination, and selected tab → **the URL** (`searchParams`), read and written through one typed, zod-validated hook, giving persistence-across-navigation and shareable/refreshable links for free; forms → **React Hook Form + zod**; only genuinely cross-route UI state with no parent-child relationship → **Zustand**, and that is now almost nothing — the sidebar-collapsed flag (or `localStorage`) and the transient multi-select set used for Compare, essentially nothing else. Form state, tab selection, single-component modal open/closed state, and table sort/filter/pagination state are all explicitly out of Zustand's scope. The current `sessionStorage`/`localStorage` filter-juggling and any hand-rolled global store (e.g. a `useSyncExternalStore`-based chat store) are deleted, not carried forward.

**Data types are derived, never hand-written**: every exported frontend domain type comes from its canonical Zod schema via `z.infer`, so a type cannot drift from what feature code is allowed to consume. Private wire schemas describe backend payloads as they exist today and may use snake_case, legacy names, or explicit unions of verified variants; their inferred types never escape the feature's `api/` folder. The current 1,100-line standalone contracts file is split per feature and replaced by schema-derived canonical types.

**Auth and tenant scope**: a small, genuinely-global auth context holds `session`, `user`, `memberships`, `isPlatformAdmin`, `currentTenant`, selected tenant id, and scope mode — rebuilt much smaller than the current implementation, without its redundant state-syncing and manual token juggling. The two persisted preferences form one concrete auth-preferences record validated as an exact versioned Zod schema; no generic storage abstraction or raw `localStorage` trust. One hook exposes the resolved tenant id set and a stable `scopeKey` string. **Every feature's React Query keys begin with that `scopeKey`** (e.g. `[domain, scopeKey, ...params]`), so switching company or scope changes the key and refetches automatically — the current blunt "clear the entire query cache on any auth change" is deleted. A full cache clear is retained for **sign-out only**, which is a security requirement (one user must never see another user's cached data), not a convenience.

**Route-level authorization** (a real security gap today — every `/admin/*` route is registered with no role check, so a normal logged-in user who types an admin URL reaches the page; only the nav link is hidden): three guarded route groups — **public** (`/careers/*`, no auth), **authenticated** (the recruiter app, redirect to sign-in if no session), and **admin** (`/admin/*`, redirect or 403 if not admin). Guards are UI defense only; the backend must still enforce its own rules. The "access pending" state (logged in, no membership, not admin) is its own screen, not an error.

**API transport — all data flows through Edge Functions**: the frontend stops reading database tables directly (the current mix of `functions.invoke`, direct `.from(...).select()` table reads, storage-signed-URL branching, and inline auth-lifecycle calls is collapsed to one call shape). Every data request goes through an Edge Function so there is exactly one boundary to validate and one place to mock in tests. The transport is one small file wrapping the function-invoke call and normalizing the error shape — nothing domain-specific, and it attaches nothing extra since the Supabase client already sends the auth token. Where a needed Edge Function does not yet exist, that is flagged as a backend task — a direct table read is never re-added as a workaround. Auth state, being a stateful client that cannot be cleanly intercepted at the network level, is deliberately _not_ routed through this seam.

**No silent fallbacks**: every multi-path fallback chain and every catch-block-that-returns-empty-or-default is deleted. One request, one path. A failure surfaces through React Query's error state as a friendly message plus a Retry that calls `refetch()`; genuinely empty data shows an empty state; a malformed response throws (see the API-layer decision). A real outage must never be indistinguishable from "no data."

**Charts / data visualization**: the hand-rolled SVG/CSS charts on the insights pages are replaced by **Recharts via shadcn's `chart` component** (which reads the `--chart-1..8` tokens from the design-token decision). Each chart gets a text alternative or accessible label so it is not invisible to screen readers. A lower-level library (visx/d3) is reached for only if a specific chart proves impossible in Recharts.

**Performance & loading**: every route is lazy-loaded (dynamic import + a route-level suspense fallback) so the first load is small; the current eager import of all ~30 pages is dropped. Loading UI is React Query's pending state plus skeleton components (the current skeleton style is kept). Suspense is used for route code-splitting only, not for data fetching (which would complicate the error boundaries). Icons and components are imported directly, not through barrel files, to keep bundles small.

**Public careers pages — SEO/metadata** (pragmatic default, revisited only if organic job traffic becomes a goal): the public surface stays a client-side SPA — no SSR/Next.js. For the two public routes only: per-page `<title>` + meta description, JSON-LD `JobPosting` structured data on the job-detail page (what Google for Jobs reads, obtainable without SSR), and a build-generated `sitemap.xml` of the public job URLs. If real SEO later becomes a priority, the follow-up is to prerender only those two routes at build time, not to rewrite the app.

**File / binary flows** (these do not fit the "get JSON, parse with zod" hook shape): signed file URLs come from an Edge Function that returns a plain URL string — the backend owns any storage-provider branching, the frontend just opens the string; CV uploads use a mutation hook posting multipart with visible progress and clear success/error; CSV export is split into a pure, unit-tested `toCsv(rows)` function plus a feature-local `downloadCsv()` browser operation, replacing the current inline blob/anchor code without creating a speculative generic blob helper.

**Screen inventory & cutover**: before porting, a keep/drop/later table is built and each row verified against the real backend. Confirmed dead code (unreferenced, unrouted) is deleted, not ported. Capabilities backed only by fake data today (with no real endpoint) are marked "later," not rebuilt against a mock. "v2 is ready" is defined by an explicit checklist: all "keep" routes work against the real backend; brand/visual parity holds; lint, test, and build are green with no `any` and no unjustified `as`; the clean-URL rewrite rule is in place. While v2 is built, feature work on the old frontend is frozen — only critical fixes ship there, each ported forward so the two don't diverge.

**Configuration & run story**: since the "run with no backend" mode is being deleted, local development runs against a real backend (local Supabase or a dev project) — no browser mock is shipped by default (an optional browser-MSW mode behind an explicit flag may be added later if offline work becomes painful, but not now). A README and `.env.example` document the required `VITE_*` variables, and the known trap that the env file lives at the repo root (the Vite `envDir` points one level up from the frontend folder) is documented explicitly because it has caused problems before. Dead flags (e.g. the insights-platform toggle whose fallback branch is being removed) are deleted. If required config is missing in dev, a clear "app is not configured" screen shows — never a silent fallback.

**Dependency cleanup**: `framer-motion` (in `package.json`, imported nowhere) is removed — animation uses `tailwindcss-animate` (ships with shadcn) plus CSS, with a motion library re-added only if a specific animation genuinely needs it. The second lockfile is removed (npm only). Other unused dependencies are audited out during scaffold.

**Performance & React coding standards — the `react-best-practices` skill is mandatory.** Every `/implement` session working on `frontend-v2/` must invoke the project's `react-best-practices` skill (Vercel's React performance guide, 70 rules) at least once and follow it. It is a always-available reference — its content is not copied into this spec or the tickets; the agent reads it from the skill. Two standing clarifications reconcile it with this spec, checked for contradictions and found compatible: (1) where the skill's `client-swr-dedup` rule names **SWR**, this project uses **React Query** instead — same principle (automatic request dedup/caching), and React Query is the decided tool; the agent must not switch to SWR. (2) This is a client-side Vite SPA with **no server-side rendering** — the skill's "Server-Side Performance" category and its Next.js/RSC-specific rules (`next/dynamic`, `React.cache()`, server actions, hydration rules) do not apply and are skipped; where a browser equivalent exists it is used instead (e.g. the skill's `next/dynamic` maps to this project's `React.lazy` route lazy-loading). Everything else in the skill either agrees with this spec (direct imports over barrel files, versioned/minimal `localStorage`, lazy-loaded routes, deferred third-party scripts) or is neutral good practice to follow. Because the rewrite lives in `frontend-v2/`, the scaffold step must make this skill (and the repo's `CLAUDE.md` / `docs/agents` config) available from that directory so later sessions can actually invoke it.

**API transport and compatibility adapters**: the current single, flat, 51-method interface spanning roughly ten unrelated business domains is not preserved. `lib/api/client.ts` is a small domain-blind transport that attaches auth, invokes an Edge Function, normalizes transport errors, and returns successful JSON as `unknown`; it has no generic `<T>` escape hatch, recursive key-casing converter, domain schema, or fallback value. Each feature owns its current backend knowledge under `features/<domain>/api/`: private **wire schemas** describe verified response variants, explicit **compatibility adapters** parse and transform them, canonical camelCase schemas define the frontend model, and request encoders translate canonical form/command input into the backend's current request keys. React Query query functions call those adapters and return canonical models, so raw payloads never enter the query cache; React Query `select` may derive a view from canonical data but must not perform wire validation or normalization.

Compatibility is deliberate rather than hopeful. Before replacing a current mapper, the implementing ticket inventories every response alias/fallback chain it owns (for example `current_title ?? currentTitle ?? job_title ?? title` and `years_experience ?? yearsExperience ?? total_experience`), verifies each accepted variant against the producing Edge Function/RPC or captured real response, and records its precedence and null/absence semantics. A single endpoint with multiple real legacy shapes uses an explicit Zod union or a small named resolver followed by `.pipe(CanonicalSchema)`; hypothetical aliases are rejected. If two aliases that mean the same canonical field are simultaneously present with different non-null values, parsing fails instead of silently choosing one. Required and security-sensitive fields never receive defaults or broad `String()`/`Number()` coercion. Product-approved display fallbacks happen after parsing in presentation code, and genuinely optional data is modeled as optional or nullable.

Every accepted wire variant has a fixture-based adapter test asserting the exact canonical output. Tests also cover missing required fields, invalid enum/range values, null semantics, and conflicting aliases; the feature's MSW test proves the raw network fixture becomes canonical query data. This is the coverage mechanism for today's defensive hedging: Zod enforces the cases in the inventory, while the inventory and fixtures prove which cases were intentionally carried forward. When the backend is cleaned later, only the private wire schemas, transforms, request encoders, and their fixtures change; canonical schemas, query keys, hooks, forms, and components stay stable. Mock-vs-real switching remains entirely at the network level, and backend-side transport fallback logic is not reproduced in the frontend.

**Error handling & observability**: two tiers of error boundaries — one wrapping the entire application/router (catastrophic-failure fallback) and one per route/page (so a failure in one section doesn't take down shared navigation/chrome). A blocking error (a page or section that failed to load) renders inside its boundary's fallback using a consistent, destructive-styled alert component; a transient error (an action failed but the page is still usable) surfaces as a toast notification. In both cases, a message-mapping layer decides what's shown: known, expected error conditions get specific, purpose-written user-facing copy; anything else falls through to one generic message, with the underlying error logged rather than displayed. Raw exception text (`error.message`) and raw schema-validation failures are never shown to a user under any circumstance — this replaces the current pattern found in multiple places across the app. Adding a dedicated error-reporting/observability tool (e.g. for aggregating and alerting on production errors) is explicitly deferred, not part of this rewrite; the error-handling structure above is designed so that adding one later is a small, localized addition rather than a redesign.

**Routing & recruiter/candidate separation**: the app is not split into two separate deployable applications — the current single-app structure is retained, with the router's existing separation between the small public surface and the much larger internal/recruiter surface made explicit rather than implicit. The public/candidate-facing surface (job browsing and application) is organized under its own clearly, unambiguously named feature area, distinct from and sharing no naming with the recruiter-facing "candidate data record" domain — the term used for the recruiter-side data entity is reserved exclusively for that meaning throughout the codebase. Routing switches from hash-based URLs to standard clean URLs; this requires a corresponding server-side catch-all rewrite rule at actual deployment time (the current hash-routing choice exists specifically to avoid needing one), which is a required companion change for whoever executes the eventual deployment cutover, not an implicit assumption.

## Testing Decisions

Feature integration tests exercise real behavior through one network seam: components and pages render through React Testing Library exactly as they would in the browser, with backend responses supplied by MSW rather than mocked hooks or client functions. Pure compatibility-adapter tests additionally exercise the adapter's public parse function directly with raw fixtures; this is validation of the seam itself, not another runtime mocking strategy.

There is one honest exception to "only the network is mocked," because not everything the app touches is a network call: **auth is not intercepted via the network seam**. Auth is a stateful client (session, token refresh, password recovery) that MSW cannot cleanly intercept, so tests render through a shared provider that injects a fake session and memberships. The testing ticket states this explicitly: one network seam (MSW) for data; auth state injected at the provider. Beyond auth and the network, tests also touch browser storage, file download, timers, env, and `matchMedia`, so a single shared test kit is provided as those real needs appear: a setup file (jsdom, a `matchMedia` polyfill, storage reset between tests); a `renderWithProviders(ui, { route, auth })` helper wrapping Router + QueryClient + the test auth provider; an MSW server with per-feature handlers; and Vitest fake timers for debounce and cache-TTL logic. File download is made testable by construction — a pure `toCsv(rows)` function (unit-tested) plus a concrete feature-local `downloadCsv()` operation — rather than asserting on a real anchor-element click.

A good test in this codebase verifies externally observable behavior (what a user sees or what a consumer of a module receives) and never asserts on internal implementation details (state variable names, internal function call counts, component internals). Business logic — Zod schemas, API-layer functions, custom hooks, and any state stores — is the priority for test coverage as it's written; purely presentational components without real conditional logic generally don't need dedicated tests. Add the smallest test that protects a real contract or regression; do not add tests merely to mirror implementation structure. There is no enforced coverage percentage; coverage is a convention applied as code is written; this mirrors how `/tdd` already expects feature work to be tested going forward, named explicitly here rather than left implicit.

There is no prior art for this in the current codebase — zero test files exist today (`npm test` runs a non-framework custom script) — so this establishes the testing pattern for the project from scratch rather than extending an existing one.

## Out of Scope

- **The backend** (`supabase/functions/`, RPCs, database schema/migrations). Acknowledged as needing its own rewrite eventually, but that is a separate future effort; this rewrite's frontend API layer is deliberately designed to stay pluggable against whatever the backend becomes later, not to fix backend architecture now.
- **Splitting into two separate deployable applications.** Considered and explicitly rejected — the public candidate-facing surface is 2 of 20+ routes, not large enough to justify separate infrastructure; the fix is naming/folder clarity within one app.
- **An error-reporting/observability tool** (Sentry or similar). Deferred, not part of this rewrite — the app is pre-launch with no error volume yet to justify it now.
- **The outer-repo CI pipeline.** This rewrite covers the frontend project's own tooling (lint, test, format) only, not how or whether it runs in CI.
- **Internationalization / multi-language support.** Not yet understood well enough to scope — revisit once it's clear whether the product actually needs it.
- **Unrelated bug fixing in the old frontend.** Already-understood problems that matter to v2 (raw error text, missing image `alt`, unstable list keys, duplicate lockfiles, missing README, and native destructive confirmation) are captured by the build tickets and standing rules. Fixing the old implementation in place remains out of scope.
- **Backend-dependent capabilities, flagged not faked.** Because the frontend stops reading tables directly and routes everything through Edge Functions, any capability currently served only by a direct table read (candidate detail, Manatal sync counts, workspace stats, filter facets, storage signed URLs) needs a corresponding Edge Function to exist; where one doesn't, that is a backend task, flagged — never worked around with a re-added direct table read or a fake fallback. Separately, capabilities that are fake-only today with no real endpoint at all (analytics, data connectors, indexing workbench, access roster) are marked "later" and not rebuilt against a mock. Deciding or building those backend endpoints is out of scope for this frontend rewrite.
- **SSR / server-side rendering** for the public careers pages. The SEO work is limited to client-side metadata, JSON-LD, and a build-time sitemap; a prerender step for the two public routes is a possible future follow-up, not part of this rewrite.
